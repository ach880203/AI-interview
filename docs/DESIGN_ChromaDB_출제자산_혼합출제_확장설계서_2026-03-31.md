# ChromaDB 출제 자산 혼합 출제 확장 설계서

작성일: 2026-03-31

## 1. 문서 목적

이 문서는 현재 프로젝트에 이미 연결된 ChromaDB를 `문서 RAG 저장소`에서 한 단계 확장하여, `AI가 생성한 학습 문제`와 `AI가 생성한 면접 질문`도 함께 저장하고 이후 출제 시 재사용할 수 있도록 설계하는 문서이다.

목표는 아래 2가지를 동시에 만족하는 것이다.

1. 기존 문서 RAG 기능은 그대로 유지한다.
2. 이후 출제는 `크로마DB 재사용 1 : AI 신규 생성 2` 비율의 혼합 방식으로 발전시킨다.

---

## 2. 현재 상태 요약

현재 ChromaDB는 아래 역할만 수행하고 있다.

- 이력서, 자기소개서, 채용공고를 청크로 분할해 벡터 저장
- 면접 질문 생성 시 관련 문서 청크를 검색하여 RAG 컨텍스트로 주입
- 컬렉션 단위는 `session_{session_id}`
- 세션 종료 시 컬렉션 삭제

즉 현재 구조는 `출제 자산 저장소`가 아니라 `세션별 문서 검색 저장소`이다.

이 구조는 아래 목적에는 적합하다.

- 사용자 문서 기반 맞춤 질문 생성
- 임시 세션 RAG

하지만 아래 목적에는 적합하지 않다.

- 과거에 생성한 문제/질문의 누적 재사용
- 고품질 문제 라이브러리 구축
- 사용자별 중복 제어
- `크로마DB 1 : AI 생성 2` 혼합 출제

---

## 3. 확장 목표

### 3.1 핵심 목표

- AI가 생성한 문제와 질문을 별도 컬렉션에 영구 저장한다.
- 저장된 출제 자산을 검색하여 일부를 재사용하거나 변형 출제에 사용한다.
- 부족한 수량은 AI가 신규 생성한다.
- 최종적으로 `재사용 1 : 신규 생성 2` 비율을 기본값으로 운영한다.

### 3.2 확장 이후 기대 효과

- 출제 속도 개선
- OpenAI 호출 비용 절감
- 검증된 문제 재활용
- 과목/직무별 출제 품질 안정화
- 약점 기반 심화학습 품질 향상

---

## 4. 아키텍처 방향

### 4.1 저장소 역할 분리

ChromaDB 안에서 역할을 분리해야 한다.

1. 문서 RAG 저장소
- 목적: 이력서, 자소서, 채용공고 검색
- 특징: 세션 단위 임시 저장, 세션 종료 시 삭제 가능

2. 출제 자산 저장소
- 목적: 문제/질문 재사용 및 변형 출제
- 특징: 영구 저장, 누적 품질 관리, 재사용 통계 관리

즉 `문서 컬렉션`과 `출제 자산 컬렉션`은 반드시 분리한다.

### 4.2 출제 방식 변화

현재:

- 문서 검색
- AI가 전부 신규 생성

확장 후:

- 문서 검색
- 출제 자산 검색
- 일부는 기존 자산 재사용 또는 변형
- 나머지는 AI 신규 생성

## 5. 컬렉션 구조

### 5.1 유지 컬렉션

#### `session_{session_id}`

용도:
- 면접 세션용 문서 RAG

저장 대상:
- resume 청크
- cover_letter 청크
- job_description 청크

수명:
- 세션 종료 시 삭제 가능

비고:
- 현재 구현 유지

### 5.2 신규 컬렉션

#### `learning_problem_assets`

용도:
- AI가 생성한 학습 문제 자산 저장
- 재사용/변형/유사 문제 검색

저장 대상:
- 문제 본문
- 정답
- 해설
- 과목/난이도/유형/개념 태그

수명:
- 영구 저장

#### `interview_question_assets`

용도:
- AI가 생성한 면접 질문 자산 저장
- 직무/질문 유형/문맥 기반 재사용

저장 대상:
- 질문 본문
- 질문 유형
- 직무/키워드
- 사용 문맥 요약

수명:
- 영구 저장

#### 선택 컬렉션: `learning_concept_assets`

용도:
- 문제 자체보다 `개념 중심` 검색 품질을 높이고 싶을 때 사용

저장 대상:
- 개념 설명
- 대표 키워드
- 과목/난이도
- 연결된 문제 ID 목록

비고:
- 1차 구현에서는 생략 가능
- 2차 고도화 때 추천

### 5.3 컬렉션 분리 이유

- 세션 삭제 로직이 출제 자산까지 지우면 안 된다.
- 문서 청크와 문제/질문 자산은 검색 목적이 다르다.
- 메타데이터 구조가 다르다.
- 운영 시 품질 관리 기준도 다르다.

## 6. 메타데이터 설계

### 6.1 공통 메타데이터

모든 출제 자산 컬렉션에는 아래 공통 메타데이터를 둔다.

| 필드명 | 설명 |
|---|---|
| `asset_id` | 자산 고유 ID |
| `asset_type` | `learning_problem` 또는 `interview_question` |
| `source` | `ai_generated`, `ai_rewritten`, `human_curated` |
| `quality_score` | 내부 품질 점수 |
| `reuse_count` | 재사용 횟수 |
| `created_at` | 생성 시각 |
| `last_used_at` | 마지막 사용 시각 |
| `active` | 재사용 가능 여부 |
| `version` | 자산 버전 |
| `language` | 기본 `ko` |

### 6.2 학습 문제 메타데이터

`learning_problem_assets`에는 아래 필드를 추가한다.

| 필드명 | 설명 |
|---|---|
| `subject_id` | 과목 ID |
| `subject_name` | 과목명 |
| `difficulty` | `EASY`, `MEDIUM`, `HARD` |
| `problem_type` | `MULTIPLE`, `SHORT` |
| `concept_tags` | 핵심 개념 태그 목록 |
| `question_text` | 문제 본문 |
| `answer_text` | 정답 |
| `explanation_text` | 해설 |
| `choice_count` | 객관식 보기 수 |
| `generation_prompt_version` | 생성 프롬프트 버전 |
| `correct_rate` | 실제 사용자 정답률 |
| `ambiguity_flag` | 애매한 문제 여부 |
| `banned_for_reuse` | 재사용 금지 여부 |

### 6.3 면접 질문 메타데이터

`interview_question_assets`에는 아래 필드를 추가한다.

| 필드명 | 설명 |
|---|---|
| `job_family` | 직무군 |
| `question_type` | `INITIAL`, `FOLLOWUP`, `BEHAVIORAL`, `TECHNICAL` 등 |
| `difficulty` | 질문 난이도 |
| `resume_keywords` | 이력서 핵심 키워드 |
| `job_keywords` | 채용공고 핵심 키워드 |
| `conversation_stage` | 면접 초반/중반/심화 등 단계 |
| `question_text` | 질문 본문 |
| `context_summary` | 생성 당시 문맥 요약 |
| `followup_target` | 꼬리질문 대상 주제 |
| `user_response_quality_avg` | 평균 답변 품질 추정값 |
| `banned_for_reuse` | 재사용 금지 여부 |

### 6.4 메타데이터 저장 원칙

- ChromaDB에는 검색용 핵심 메타데이터만 저장한다.
- 상세 로그와 운영 통계는 MariaDB에도 함께 저장하는 것을 권장한다.
- 이유는 ChromaDB는 검색에 강하고, 관계형 집계는 MariaDB가 더 적합하기 때문이다.

---

## 7. 저장 단위 설계

### 7.1 학습 문제 저장 단위

학습 문제는 아래 2개를 함께 저장한다.

1. 벡터 검색용 문서 본문
- 추천 값: `question + explanation + concept summary`

2. 메타데이터
- 과목, 난이도, 문제유형, 정답률, 개념 태그

권장 이유:
- 문제 본문만 저장하면 의미 검색이 약하다.
- 해설과 개념 요약을 함께 넣어야 유사 개념 검색 품질이 좋아진다.

### 7.2 면접 질문 저장 단위

면접 질문은 아래 조합으로 저장한다.

- 질문 본문
- 질문 생성 당시 핵심 문맥 요약
- 직무 키워드
- 질문 유형

권장 이유:
- 면접 질문은 단순 문장 유사도보다 `문맥 유사도`가 중요하다.
- 이력서 키워드와 직무 키워드를 같이 저장해야 검색 품질이 올라간다.

## 8. 혼합 출제 알고리즘 설계

## 8.1 공통 원칙

- 기본 목표 비율은 `재사용 1 : 신규 생성 2`
- 단, 품질이 낮거나 충분한 후보가 없으면 비율을 강제하지 않는다.
- `항상 품질 우선`, `비율은 보조 정책`으로 운영한다.

기본 원칙:

1. 후보 검색
2. 중복 제거
3. 품질 필터링
4. 목표 수량만큼 재사용/변형 선택
5. 부족분 신규 생성
6. 최종 세트 다양성 검사

### 8.2 학습 문제 혼합 출제 알고리즘

입력값 예시:

- 과목
- 난이도
- 문제 유형
- 사용자 정답률
- 최근 오답 개념
- 목표 문제 수

기본 흐름:

1. `learning_problem_assets`에서 검색한다.
2. 조건 우선순위는 아래와 같다.
- 같은 과목
- 같은 난이도
- 같은 문제 유형
- 최근 오답 개념 또는 약점 개념 일치

3. 검색된 후보를 필터링한다.
- 최근 해당 사용자에게 노출한 문제 제거
- `banned_for_reuse=true` 제거
- `ambiguity_flag=true` 제거
- `quality_score` 하한선 미달 제거

4. 남은 후보 중 목표 수량의 1/3을 선택한다.
- 예: 총 6문제면 2문제

5. 선택한 2문제는 아래 중 하나로 사용한다.
- 그대로 재사용
- 보기/문장만 바꾼 변형 문제로 재작성

6. 나머지 4문제는 AI가 신규 생성한다.
- 이때 검색된 기존 문제의 개념 태그를 프롬프트에 넣어
- 품질과 일관성을 높인다.

7. 최종 세트를 검사한다.
- 같은 개념만 과도하게 몰리지 않는지 확인
- 객관식/주관식 비중 확인
- 정답 패턴 편향 확인

### 8.3 학습 문제 혼합 출제 의사결정 규칙

권장 기준:

- 후보가 3개 미만이면 전부 신규 생성
- 후보가 충분하지만 중복 위험이 높으면 `재사용 기반 변형` 우선
- 사용자가 최근 틀린 개념이면 재사용 비중을 일시적으로 40%까지 올릴 수 있음

### 8.4 면접 질문 혼합 출제 알고리즘

입력값 예시:

- session_id
- question_type
- 직무 정보
- 이력서 키워드
- 채용공고 키워드
- 대화 이력

기본 흐름:

1. 문서 RAG는 기존대로 수행한다.
2. 동시에 `interview_question_assets`에서 유사 질문 후보를 검색한다.
3. 후보 검색 조건은 아래와 같다.
- 같은 직무군
- 같은 질문 유형
- 비슷한 이력서/채용공고 키워드
- 비슷한 대화 단계

4. 후보를 필터링한다.
- 같은 세션에서 이미 쓴 질문 제외
- 사용자 과거 이력에서 최근 반복 노출 질문 제외
- 재사용 금지 질문 제외

5. 후보 중 1개를 선택한다.
- 그대로 쓰거나
- 문맥에 맞는 꼬리 질문으로 변형한다.

6. 나머지 2개 분량의 역할은 AI 신규 생성에 준다.
- 실제 운영에서는 질문을 한 번에 3개 보여주는 것이 아니라
- 내부 생성 풀 기준으로 1개는 재사용/변형, 2개는 신규 생성 후
- 최적 1개를 선택하는 방식도 가능하다.

### 8.5 면접 질문 혼합 출제 권장 방식

면접은 학습 문제보다 중복 체감이 더 크므로 아래 방식이 더 좋다.

- `직접 재사용`: 20%
- `재사용 기반 변형`: 20%
- `완전 신규 생성`: 60%

즉 사용자가 체감하기에는 새롭지만, 내부적으로는 기존 고품질 질문 자산을 활용하는 구조가 좋다.

## 9. 재사용 방식 정책

### 9.1 직접 재사용

정의:
- 저장된 문제/질문을 거의 그대로 사용

장점:
- 빠름
- 비용 적음
- 검증된 자산 활용 가능

단점:
- 중복감이 큼

권장 대상:
- 개념 점검용 문제
- 대표 면접 질문

### 9.2 재사용 기반 변형

정의:
- 기존 자산을 시드로 삼아 표현, 보기, 맥락을 바꿔 재작성

장점:
- 품질 안정성과 신규성의 균형이 좋음

단점:
- 구현 복잡도 상승

권장 대상:
- 심화학습 문제
- 후속 꼬리 질문

### 9.3 완전 신규 생성

정의:
- 기존 자산 검색 없이 AI가 새로 생성

장점:
- 다양성 최고

단점:
- 비용 증가
- 품질 편차 가능

권장 대상:
- 새로운 직무/새로운 개념
- 검색 후보가 부족한 경우

## 10. API 변경안

### 10.1 학습 문제 저장 API

#### `POST /vector/assets/learning-problems`

목적:
- 생성된 학습 문제를 ChromaDB 자산 컬렉션에 저장

요청 예시 필드:

- `subjectId`
- `subjectName`
- `difficulty`
- `problemType`
- `question`
- `choices`
- `answer`
- `explanation`
- `conceptTags`
- `source`
- `qualityScore`

응답:
- 저장된 `assetId`
- 저장 성공 여부

### 10.2 면접 질문 저장 API

#### `POST /vector/assets/interview-questions`

목적:
- 생성된 면접 질문을 ChromaDB 자산 컬렉션에 저장

요청 예시 필드:

- `jobFamily`
- `questionType`
- `difficulty`
- `question`
- `resumeKeywords`
- `jobKeywords`
- `contextSummary`
- `source`
- `qualityScore`

응답:
- 저장된 `assetId`
- 저장 성공 여부

### 10.3 학습 문제 검색 API

#### `POST /vector/assets/learning-problems/search`

목적:
- 과목/난이도/개념 기준으로 재사용 후보 검색

요청 예시 필드:

- `subjectId`
- `difficulty`
- `problemType`
- `conceptTags`

## 11. 서비스 로직 변경안

### 11.1 학습 생성 로직 변경

현재:
- AI가 문제 전부 생성

변경 후:

1. 재사용 후보 검색
2. 목표 수량의 1/3 확보
3. 부족분 AI 신규 생성
4. 최종 문제 세트 반환
5. 신규 생성 문제는 다시 자산 컬렉션에 저장

### 11.2 면접 질문 생성 로직 변경

현재:
- 문서 RAG만 사용하고 질문은 AI가 생성

변경 후:

1. 문서 RAG 유지
2. 질문 자산 검색 추가
3. 재사용 또는 변형 후보 1개 확보
4. 신규 생성 후보와 비교
5. 최종 질문 선택
6. 새 질문은 자산 컬렉션에 저장

## 12. 품질 관리 정책

### 12.1 저장 차단 기준

아래 조건이면 자산 저장을 막거나 보류한다.

- 정답/해설이 비어 있음
- 질문/문제 문장이 지나치게 짧음
- 중복률이 높음
- 채점 결과가 자주 흔들림
- 사용자 반응이 매우 나쁨

### 12.2 재사용 금지 기준

- 정답률이 지나치게 낮아 애매한 문제로 판단됨
- 해설 품질이 낮음
- 오탈자/문장 오류가 발견됨
- 면접 질문이 특정 직무 문맥에서 부적절함

### 12.3 품질 점수 산정 예시

초기 품질 점수는 아래 요소를 기준으로 부여할 수 있다.

- 생성 직후 구조 완성도
- 해설 충실도
- 중복 여부
- 실제 사용자 정답률
- 재사용 후 만족도

## 13. 중복 방지 정책

### 13.1 사용자 기준 중복 방지

- 최근 N일 이내에 같은 자산 노출 금지
- 같은 세션에서는 동일 자산 재노출 금지
- 유사도 임계치 이상인 문제도 중복 후보로 간주

### 13.2 콘텐츠 기준 중복 방지

- 같은 개념이 연속 2문제 이상 반복되지 않도록 제어
- 같은 정답 패턴 반복 최소화
- 면접 질문은 같은 키워드만 계속 파고들지 않도록 다양성 검사

## 17. 위험 요소와 대응

### 위험 1. 중복 체감 증가

대응:
- 직접 재사용보다 변형 재사용 비중 확대
- 사용자 최근 노출 이력 제외

### 위험 2. 품질 낮은 자산 누적

대응:
- 저장 시 최소 품질 기준
- 운영 중 재사용 금지 플래그 도입

### 위험 3. 검색 결과는 유사하지만 실제로는 부적절

대응:
- 메타데이터 필터를 먼저 적용한 뒤 벡터 검색
- 벡터 검색만 단독으로 쓰지 않음

### 위험 4. 비율 강제 때문에 품질 하락

대응:
- `1 : 2`는 목표값일 뿐 강제값이 아님
- 후보가 부족하면 전부 신규 생성 허용

## 18. 최종 결론

이 프로젝트는 이미 ChromaDB가 연결되어 있으므로, 구조만 올바르게 분리하면 `문서 RAG + 출제 자산 재사용` 모델로 충분히 확장 가능하다.

가장 중요한 원칙은 아래 4가지다.

1. 문서 컬렉션과 출제 자산 컬렉션을 분리한다.
2. 문제/질문 본문만 저장하지 말고 메타데이터를 함께 저장한다.
3. `재사용 1 : 신규 2`는 품질 우선 조건에서 적용한다.
4. 직접 재사용보다 `재사용 기반 변형`을 적극 활용한다.

이 설계대로 진행하면 크로마DB는 단순 문서 검색기를 넘어서, `학습 문제와 면접 질문의 품질을 누적시키는 자산 저장소` 역할까지 수행할 수 있다.
