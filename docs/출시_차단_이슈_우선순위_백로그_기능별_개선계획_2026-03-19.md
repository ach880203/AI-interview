# 출시 차단 이슈, 우선순위 백로그, 기능별 개선 계획

## 1. 문서 목적

이 문서는 이전 진단 보고서인 [프로젝트_종합_실행_진단_보고서_2026-03-19.md](C:/Programmer/Work/AI-interview/docs/프로젝트_종합_실행_진단_보고서_2026-03-19.md)를 이어서,
프론트 빌드 이슈를 제외한 나머지 항목만 별도로 정리한 실행 계획 문서입니다.

이번 문서의 목표는 아래 3가지입니다.

1. 프론트를 제외하고도 당장 출시를 막는 이슈가 무엇인지 명확히 구분한다.
2. 무엇부터 처리해야 하는지 우선순위 백로그로 정리한다.
3. 기능별로 어떤 수준까지 정리해야 출시 가능한지 개선 계획으로 남긴다.

---

## 2. 이번 문서의 전제

- 프론트엔드 빌드 오류 평가는 이번 문서에서 제외합니다.
- 코드 수정은 하지 않고, 현재 상태를 기준으로 계획만 정리합니다.
- 기준은 "데모 공개"가 아니라 "사용자가 실제로 써도 되는 수준의 출시"입니다.

즉, 이번 문서에서는 "프론트만 고치면 끝"이라는 관점으로 보지 않습니다.  
프론트가 정리되더라도, 아래 항목들이 해결되지 않으면 여전히 출시 리스크가 큽니다.

---

## 3. 결론 먼저

현재 상태에서 프론트를 제외하고도 바로 출시를 막는 핵심 이슈는 아래 5가지입니다.

1. 학습 문제 생성이 실제 실행에서 실패한다.
2. 학습 과목 데이터가 중복 누적돼 데이터 신뢰가 깨진다.
3. 인증 API 주소 체계가 배포 환경 기준으로 불안정하다.
4. 결제와 파일 저장이 실제 운영 흐름이 아니라 Mock 중심이다.
5. 마이페이지/구독 등 일부 화면이 아직 준비중인데 실제 메뉴로 노출돼 있다.

이 다섯 가지 중 특히 1, 3, 4, 5는 출시 판단을 직접 막는 항목입니다.

## 5. 우선순위 백로그

### 5-1. 이번 릴리스 전에 반드시 처리할 백로그

| 우선순위 | 백로그 | 목표 | 완료 기준 |
|---|---|---|---|
| 1 | 학습 문제 생성 오류 해결 | 학습 기능이 실제로 끝까지 돌아가게 한다 | 문제 생성, 채점, 통계까지 한 흐름이 실제 API로 통과 |
| 2 | 인증 주소 체계 통일 | 배포 환경에서도 로그인/로그아웃/토큰 갱신이 안정적으로 동작하게 한다 | 인증 관련 요청이 한 기준 URL만 사용 |
| 3 | 준비중 페이지 노출 정책 정리 | 연결 안 된 기능이 사용자 앞에 노출되지 않게 한다 | 비완성 메뉴 숨김 또는 진입 차단 정책 적용 |
| 4 | 결제/저장소 기능 출시 범위 확정 | 진짜 운영할 기능과 데모용 기능을 분리한다 | 운영 기능과 비공개 기능 범위가 문서와 화면에 일치 |
| 5 | 학습 과목 중복 정리 | 과목 목록과 통계 신뢰를 회복한다 | 중복 없는 과목 목록과 재실행 안전성 확보 |

### 5-2. 릴리스 직후 안정화 백로그

|---|---|---|---|
| 6 | 한글 문구 전수 정리 | 전 화면 한국어 일관성을 맞춘다 | 사용자 노출 문구 영어 제거 |
| 7 | 응답 인코딩 정책 점검 | 한글 깨짐 우려를 줄인다 | API 응답/문서/파일 저장 정책 명시 |
| 8 | 헬스체크/로그 정책 정리 | 운영 시 장애 파악 속도를 높인다 | 헬스체크 정책과 필수 로그 목록 문서화 |
| 9 | 마이페이지 실제 연결 범위 확대 | 더미 요소를 줄이고 실제 서비스 느낌을 높인다 | 문의/프로필/찜 등 노출 기능의 실제 연결 여부 정리 |

## 6. 기능별 개선 계획서

## 6-1. 인증 기능

### 현재 판단

인증 자체는 실제로 동작합니다.  
하지만 구조적으로는 배포 환경에서 가장 먼저 사고 날 수 있는 구간입니다.

### 현재 문제

- 공용 API와 인증 API의 기준 주소가 다릅니다.
- 로컬 주소가 직접 박혀 있어 운영 환경 대응이 약합니다.

### 출시 전 해야 할 일

1. 인증 관련 요청 URL 기준을 하나로 통일합니다.
2. 로그인, 로그아웃, 토큰 갱신 흐름을 배포 주소 기준으로 다시 점검합니다.
3. 실패 시 사용자에게 보여줄 한글 오류 문구 기준도 같이 정리합니다.

### 출시 판단 기준

- 로그인 성공
- 새로고침 후 인증 유지
- 토큰 만료 후 재발급
- 로그아웃 후 보호 페이지 차단
- 토큰의 만료를 24시간으로 제한

이 다섯 가지가 배포 환경에서 재현돼야 합니다.

---

## 6-2. 학습 기능

### 현재 판단

학습 기능은 지금 상태로는 공개 출시 불가입니다.

### 현재 문제

- 문제 생성이 실제 실행에서 실패합니다.
- 과목 데이터가 중복 누적됩니다.
- 학습이 통계와 추천으로 이어지는 신뢰가 아직 낮습니다.

### 출시 전 해야 할 일

1. AI 응답 파싱 구조를 안정화합니다.
2. 문제 생성 API의 입력/출력 형식을 고정합니다.
3. 과목 데이터 중복 방지 정책을 정합니다.
4. 문제 생성부터 채점, 통계 반영까지 통합 검증 시나리오를 만듭니다.

### 출시 판단 기준

- 과목 목록 중복 없음
- 문제 생성 성공
- 채점 성공
- 통계 반영 성공
- 실패 시 사용자에게 한글 오류 안내 제공

### 제품 관점 권고

학습 기능을 살릴 거라면 대충 붙이면 안 됩니다.  
이 기능은 면접 다음으로 서비스 신뢰를 좌우하는 축이라서, "있다"가 아니라 "잘 된다" 수준까지 끌어올려야 합니다.

의견 - 벡터데이터화 해서 크로마DB를 이용 문제를 ai를 통하여 각 각 500~1000문제 가량을 문제와 답과 짧은 해설을 생성하여 데이터로 심어두고, 부족한 부분만 ai가 생성해주고,
       더 알고 싶은 부분이 있다면 ai심화학습으로 넘어가게 기능을 줄 수 는 없는지 의견을 내주고 가능하다면 계획을 수립해줘.

## 6-3. 면접 기능

### 현재 판단

현재 프로젝트에서 가장 경쟁력 있는 기능입니다.

### 현재 문제

- 결과는 나오지만, 왜 이런 점수인지 설명력이 더 필요합니다.
- 음성 저장/재생/추적은 아직 운영형이 아닙니다.

### 출시 전 해야 할 일

1. 면접 흐름은 지금처럼 핵심 기능으로 유지합니다.
2. 결과 화면에서 점수 근거, 강점, 약점, 다음 액션을 더 구조적으로 보여줍니다.
3. 음성 기능은 운영형으로 갈지, MVP에서는 텍스트 중심으로 갈지 범위를 정합니다.

### 출시 판단 기준

- 세션 시작
- 질문 진행
- 답변 제출
- 종료
- 결과 확인

이 흐름이 반복 검증에서 안정적으로 유지돼야 합니다.

### 제품 관점 권고

이 프로젝트는 지금 시점에서 커머스보다 면접이 훨씬 강합니다.  
따라서 릴리스 메시지도 "면접 훈련 서비스"를 중심으로 잡는 편이 맞습니다.

의견 - 이것도 학습과 마찬가지로 기본적인 기술관련 질문들은 벡터데이터화 해서 크로마db를 이용해서 속도감있게 불러오고, 이력서, 자소서, 공고의 관련된 질문은 ai를 통해서
        전문적이고 관련된 질문을 생성해주는 방식으로 가능한지 의견을 내줘.

## 6-4. 문서 관리 기능

### 현재 판단

포트폴리오용 구조는 좋지만, 운영형 문서 도구 수준까지는 아직 아닙니다.

### 현재 문제

- 저장소 연동이 Mock 기반입니다.
- 파일/문서 관리가 실제 사용자 업무 흐름 기준으로는 아직 단순합니다.

### 출시 전 해야 할 일

1. 실제 출시 범위가 텍스트 저장인지 파일 업로드인지 먼저 결정합니다.
2. 파일 업로드를 유지한다면 저장소 정책을 실제화합니다.
3. 문서 편집 흐름에서 저장 실패, 교체, 삭제, 보안 처리 기준을 정합니다.

### 출시 판단 기준

- 문서 생성/수정/삭제
- 업로드 파일 저장 경로 일관성
- 실패 시 복구 또는 재시도 안내

## 6-5. 도서 / 주문 기능

### 현재 판단

지금 상태로는 상용 결제 기능으로 공개하면 안 됩니다.

### 현재 문제

- 실결제 없이 즉시 결제 완료 처리됩니다.
- 운영형 주문 관리라기보다 데모 주문 흐름에 가깝습니다.

### 출시 전 해야 할 일

1. 커머스 기능을 이번 릴리스에 포함할지 먼저 결정합니다.
2. 포함한다면 결제, 주문 실패, 환불, 재고 복구 정책을 실제화합니다.
3. 포함하지 않는다면 화면과 메뉴에서 비공개 처리합니다.

### 출시 판단 기준

- 결제 성공/실패가 실제로 분리됨
- 주문 상태 전이가 논리적으로 맞음
- 재고 차감/복구가 맞게 동작함
- 관리자에서 추적 가능함

### 제품 관점 권고

현재는 면접 서비스의 집중도를 해치고 있습니다.  
커머스를 유지할 거면 제대로 하고, 아니면 이번 릴리스에서는 과감히 빼는 편이 낫습니다.

의견 - 커머스를 유지하지 않으면 급격하게 사이즈가 줄기 때문에 무조건 유지해야하고, 결제기능도 제대로 사용할 수 있도록 결제 페이지 따로 만들어서 카카오페이결제 API만 넣으면 사용할 수 있도록 준비해주고, 구독결제 페이지를 만들었는데 거기서도 결제페이지로 잘 넘어갈 수 있도록 만들어줘. 그리고 결제는 카카오페이만 넣을거야. 그러기 위해선 주문페이지도 제대로 존재해야하니까 장바구니에서 끝내는게 아니라 주문하기 누르면 배송을 위해 주소 이름 전화번호등을 회원정보에서 받아 올 수 있게 세팅해주고, 주소는 주소API사용해서 찾을 수 있도록 API키만 넣으면 바로 동작 할 수 있게 준비해줘. 물론 홈페이지 분위기랑 느낌 잘 맞춰서 만들어 줄거라 믿어.

## 6-6. 구독 기능

### 현재 판단

지금은 공개 기능이 아니라 준비중 안내 페이지입니다.

### 현재 문제

- 결제 흐름이 없습니다.
- 사용자가 들어가도 할 수 있는 행동이 없습니다.

### 출시 전 해야 할 일

1. 실제 구독 상품과 혜택을 정의합니다.
2. 결제 수단과 구독 기간 정책을 정합니다.
3. 준비중 상태라면 메뉴에서 숨기거나 대기 신청 형태로 바꿉니다.

### 출시 판단 기준

- 상품 설명 명확
- 결제 가능
- 구독 상태 확인 가능
- 해지/변경 정책 확인 가능

이것도 제대로 결제해서 사용할 수 있게 만들자.

## 6-7. 마이페이지

### 현재 판단

구성은 넓지만 실제 서비스 연결 정도가 고르지 않습니다.

### 현재 문제

- 일부 정보는 로컬 상태/더미 데이터 기반입니다.
- 탭마다 완성도 차이가 큽니다.

### 출시 전 해야 할 일

1. 실제 연결된 탭과 아닌 탭을 구분합니다.
2. 연결되지 않은 기능은 구현하여서 명확히 처리합니다.
3. 실제 노출 기능만 남겨 사용자 신뢰를 높입니다.

### 출시 판단 기준

- 노출된 탭은 전부 실제 데이터와 연결
- 빈 상태 문구도 한글로 일관
- 저장/삭제/조회 결과가 예측 가능

의견 - 실제로 운영할 수 있는 퀄리티가 필요하니까. 운영가능한 수준으로 만들어주고, 문의하기는 관리자 페이지 문의보기 페이지랑 연결이 안됐으니까 연결해줘.

## 6-8. 운영 / 품질 / 한글 정책

### 현재 판단

기능 구현보다 늦게 챙겨진 흔적이 보입니다.

### 현재 문제

- 한글 우선 원칙이 화면 전반에 완전히 반영되지 않았습니다.
- 운영 헬스체크와 관찰성 정책이 약합니다.
- 인코딩 안정성 기준이 코드와 문서에 일관되게 보이지 않습니다.

### 출시 전 해야 할 일

1. 사용자 노출 문구 한글화 기준을 명시합니다.
2. UTF-8 저장, 응답 인코딩, JSON 처리 기준을 점검합니다.
3. 헬스체크 공개 범위와 장애 로그 기준을 정합니다.

### 출시 판단 기준

- 화면 영문 문구 최소화
- 한글 깨짐 재현 사례 없음
- 운영 점검용 URL과 로그 정책 문서 존재

## 9. 최종 판단

프론트를 제외하고 봐도, 지금 프로젝트는 아직 "전체 기능 출시" 단계는 아닙니다.

하지만 방향 자체는 나쁘지 않습니다.  
특히 면접 기능은 충분히 주력 기능으로 밀 수 있습니다.

따라서 지금 가장 좋은 판단은 아래입니다.

1. 면접 중심 MVP로 범위를 줄인다.
2. 학습 기능은 오류를 잡고 난 뒤 공개한다.
3. 커머스/구독/더미성 마이페이지 기능은 당분간 숨긴다.
4. 운영과 한글 품질 기준을 문서로 고정한다.

이렇게 가야 "범위는 조금 줄어도 완성도는 높다"는 평가를 받을 수 있습니다.
