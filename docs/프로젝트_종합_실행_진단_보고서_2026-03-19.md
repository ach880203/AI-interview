# 프로젝트 종합 실행 진단 보고서

## 1. 문서 목적

이 문서는 `AI-interview` 프로젝트를 실제로 실행해 보면서 기능을 점검한 결과를 정리한 보고서입니다.

- 코드 수정은 하지 않았습니다.
- 실제 서버를 띄워서 API 흐름을 검증했습니다.
- 프론트엔드는 빌드 검증과 소스 구조 점검으로 평가했습니다.

## 4. 치명적 문제

### 4-1. 프론트엔드가 현재 빌드되지 않습니다

이건 상용화 관점에서 바로 탈락 사유입니다. 화면이 예쁘고 기능이 좋아도, 현재 상태에서 프론트가 정상 빌드되지 않으면 배포 자체가 막힙니다.

- `frontend/src/pages/MyPage.jsx:163`
- `frontend/src/components/layout/Navbar.jsx:83`

`vite build` 결과 기준으로 JSX 파싱 오류가 발생했습니다.  
즉, "기능이 조금 덜 완성" 수준이 아니라 "현재 프론트 배포가 깨진 상태"입니다.

### 4-2. 학습 기능의 핵심인 문제 생성이 실제 실행에서 죽습니다

이 프로젝트가 면접 + 학습 플랫폼인데, 학습 문제 생성이 터지면 서비스 완성도가 크게 떨어집니다.

- `ai-server/services/learning_service.py:97`
- `backend-test-18080.out.log:797`
- `backend-test-18080.out.log:799`
- `ai-server-8000.err.log:222`
- `ai-server-8000.err.log:225`

실제 실행 결과 `POST /learning/generate` 호출 시 `AttributeError: 'str' object has no attribute 'get'` 가 발생했습니다.  
즉, AI 응답 파싱 로직이 예상한 자료형과 실제 자료형이 맞지 않습니다.

이건 단순 예외가 아니라 학습 서비스의 핵심 가치를 직접 깨는 문제입니다.

### 4-3. 학습 과목 시드 데이터가 중복 누적됩니다

과목 목록 API는 응답 자체는 성공했지만, 실제 데이터가 중복 누적되고 있었습니다.

- `backend/ai-interview/src/main/resources/data.sql:6`
- `backend/ai-interview/src/main/java/com/aimentor/domain/learning/LearningSubjectEntity.java:14`
- `backend/ai-interview/src/main/java/com/aimentor/domain/learning/LearningSubjectEntity.java:28`

`INSERT IGNORE`를 쓰고 있지만 `name` 컬럼에 유니크 제약이 없어 중복 방지가 실제로 되지 않습니다.  
실행 중 확인한 결과 과목이 2~3개씩 반복되어 내려왔습니다.

이 문제는 당장 서비스가 죽는 수준은 아니지만, 운영을 시작하면 데이터 신뢰를 빠르게 무너뜨립니다.

## 5. 중요 문제

### 5-1. 인증 API 주소가 환경 변수와 따로 놀고 있습니다

- `frontend/src/api/axios.js:10`
- `frontend/src/store/authStore.js:20`
- `frontend/src/store/authStore.js:38`
- `frontend/src/store/authStore.js:65`
- `frontend/src/store/authStore.js:86`

공용 Axios는 `VITE_API_BASE_URL`을 보지만, 인증 스토어는 `http://localhost:8080`을 하드코딩하고 있습니다.  
로컬에서는 우연히 동작해도, 배포 환경에서는 로그인만 따로 깨질 가능성이 큽니다.

이건 초반에는 잘 안 보이는데, 배포 직전에 사람을 가장 피곤하게 만드는 유형입니다.

### 5-2. 마이페이지와 구독 페이지는 아직 "완성된 기능"으로 보기 어렵습니다

- `frontend/src/pages/MyPage.jsx:40`
- `frontend/src/pages/MyPage.jsx:46`
- `frontend/src/pages/MyPage.jsx:264`
- `frontend/src/pages/MyPage.jsx:286`
- `frontend/src/pages/SubscriptionPage.jsx:28`
- `frontend/src/pages/SubscriptionPage.jsx:68`

마이페이지는 프로필 초안, 문의 내역 등 일부가 로컬 상태와 더미 데이터 중심입니다.  
구독 페이지는 결제 기능이 준비 중이며 버튼도 비활성화돼 있습니다.

즉, 화면은 있어도 "사용자가 실제로 끝까지 행동을 완료할 수 있는 페이지"는 아닙니다.

### 5-3. 주문과 파일 업로드는 아직 운영용이 아니라 데모용에 가깝습니다

- `backend/ai-interview/src/main/java/com/aimentor/domain/book/OrderService.java:23`
- `backend/ai-interview/src/main/java/com/aimentor/domain/book/OrderService.java:77`

- `backend/ai-interview/src/main/java/com/aimentor/external/storage/S3StorageService.java:48`

주문은 실결제 연동 없이 즉시 `PAID` 처리됩니다.  
파일 업로드는 S3 Mock URL 반환 방식입니다.  
면접 답변 음성도 실제 저장소/파일 흐름이 아니라 URL 자리만 잡힌 상태입니다.

포트폴리오용 데모로는 괜찮지만, 상용 서비스 기준으로는 아직 결제/저장/추적의 신뢰도가 부족합니다.

### 5-4. 한글 일관성과 인코딩 안정성에 아직 구멍이 있습니다

- `frontend/src/pages/HomePage.jsx:243`
- `frontend/src/pages/HomePage.jsx:267`
- `frontend/src/pages/HomePage.jsx:295`
- `frontend/src/pages/HomePage.jsx:346`

홈 화면에 `Core Features`, `How It Works`, `Premium Service`, `Interview Mentor` 같은 영어 문구가 남아 있습니다.  
프로젝트 규칙이 "모든 UI 문구는 한글"인데 이 기준을 아직 완전히 지키지 못했습니다.

또한 실제 응답 헤더 확인 시 `Content-Type: application/json` 까지만 내려오고 `charset=utf-8`은 명시되지 않았습니다.  
JSON 기본 인코딩 특성상 대부분의 브라우저는 처리하겠지만, "한글 깨짐 방지"를 최우선으로 둔 프로젝트라면 더 엄격하게 보는 것이 맞습니다.

### 5-5. 운영 관찰성이 부족합니다

- `backend-test-18080.out.log:71`
- `backend-test-18080.out.log:79`

`/actuator/health`가 인증 없이 바로 열려 있지 않아, 운영 환경에서 로드밸런서/모니터링 연결 시 불편해질 수 있습니다.  
보안상 일부러 막았다면 문서화가 필요하고, 아니라면 헬스체크 정책을 다시 정리해야 합니다.

## 7. 총평

이 프로젝트는 "아이디어가 빈약한 프로젝트"는 아닙니다.  
오히려 범위가 넓고, 면접/학습/문서/커머스까지 묶으려는 야심이 분명합니다.

문제는 범위를 넓게 가져간 만큼 완성도 편차가 매우 크다는 점입니다.

잘 된 부분:

- 면접 흐름은 실제로 돌아갑니다.
- 인증 기본 흐름도 살아 있습니다.
- 백엔드 테스트도 통과했습니다.
- 포트폴리오 발표에서는 설명할 거리가 많습니다.

아쉬운 부분:

- 프론트가 현재 빌드조차 안 됩니다.
- 학습 핵심 기능이 실제 실행에서 깨집니다.
- 마이페이지/구독/결제/파일 업로드는 아직 운영형이 아니라 준비형입니다.
- 한글/인코딩/문구 일관성도 프로젝트 규칙만큼 엄격하게 관리되지 않았습니다.

냉정하게 말하면, 지금 상태는 "상용 서비스 직전"이 아니라 "잘 만든 포트폴리오 데모와 미완성 기능이 섞여 있는 상태"입니다.

## 8. 점수

### 8-1. 상용 서비스 기준 총점

`45 / 100`

이 점수를 준 이유는 아래와 같습니다.

- 프론트 빌드 실패: 큰 감점
- 학습 핵심 기능 실동작 실패: 큰 감점
- 결제/저장소가 Mock 중심: 큰 감점
- 일부 페이지가 준비 중/더미 상태: 감점
- 면접 흐름 실동작, 인증 성공, 테스트 통과: 가점

즉, "가능성은 높지만 출시하면 바로 문제 터질 상태"라는 평가입니다.

### 8-2. 포트폴리오 데모 기준 총점

`71 / 100`

포트폴리오 기준 점수가 상용화 점수보다 높은 이유는,  
"실제 서비스 운영 품질"보다 "설계 범위, 기능 다양성, 발표 소재"를 더 높게 보기 때문입니다.

## 9. 발전 방향 제안

### 9-1. 1순위: 출시 차단 이슈부터 막아야 합니다

1. 프론트 빌드 오류 해결
2. 학습 문제 생성 500 오류 해결
3. 학습 과목 중복 누적 방지

이 세 가지는 먼저 정리하지 않으면 다른 개선이 다 가려집니다.

### 9-2. 2순위: "실제 서비스처럼 보이는 부분"을 진짜로 만들어야 합니다

1. 인증 API 주소를 환경 변수 체계로 통일
2. 구독 페이지를 실제 결제 흐름과 연결하거나, 아니면 과감히 숨기기
3. 마이페이지 더미 데이터 제거 및 실제 API 연결 범위 확대
4. 파일 업로드/S3/음성 저장 흐름 실제화

### 9-3. 3순위: 신뢰도를 올려야 합니다

1. 홈 화면 영어 문구를 전부 한글화
2. 하드코딩된 마케팅 수치 정리
3. 헬스체크, 운영 로그, 장애 대응 문서 정리
4. 한글 인코딩 정책을 응답/문서/화면 전반에서 더 엄격히 관리

### 9-4. 4순위: 면접 서비스 강점을 더 밀어야 합니다

이 프로젝트는 현재 커머스보다 면접 쪽이 훨씬 경쟁력이 있습니다.

그래서 우선순위는 아래가 더 좋습니다.

1. 면접 기록 비교
2. 질문 난이도 조절
3. 답변 말버릇/속도/길이 분석
4. 피드백 근거 시각화
5. 취약점 기반 맞춤 학습 연결

즉, 이것저것 다 하는 서비스보다 "면접 훈련 경험이 진짜 좋은 서비스"로 먼저 날을 세우는 편이 낫습니다.
