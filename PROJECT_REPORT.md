## 3. 아키텍처/인프라 평가

### 3.1 벡터 DB 지원 여부
**결론: 현재 벡터 DB 코드 없음**

ai-server에 LangChain 의존성이 있으나, 임베딩/벡터 저장소 코드는 전혀 없습니다.
면접 질문 생성 시 이력서/자소서 전문을 매번 프롬프트에 넣는 구조라서,
문서가 길어지면 토큰 비용과 컨텍스트 윈도우 제한에 걸립니다.

**추천 구조:**
```
이력서/자소서 업로드 → 청크 분할 → 임베딩(text-embedding-3-small)
→ ChromaDB/pgvector 저장 → 면접 질문 생성 시 유사 청크 검색(RAG)
```

### 3.2 보안 취약점
| 위험도 | 항목 | 설명 |
|--------|------|------|
| 치명 | 결제 미검증 | 클라이언트가 APPROVED를 직접 전송 → 서버가 PG 확인 없이 수락 |
| 높음 | CORS 와일드 | 개발 환경 localhost만이지만 운영에서 제한 필수 |
| 높음 | JWT Secret | 하드코딩된 기본값 존재 (application.yml 폴백) |
| 중간 | Rate Limit | AI 서버만 적용, 백엔드 API는 미적용 |
| 중간 | 파일 업로드 | 크기 제한 있으나 악성 파일 검증 없음 |
| 낮음 | XSS | React가 기본 이스케이프하지만 dangerouslySetInnerHTML 사용 확인 필요 |

### 3.3 성능 병목
- DB 쿼리 최적화 없음 (N+1 문제 가능성: 주문 항목 조회)
- 캐싱 레이어 없음 (Redis 미사용)
- 프론트엔드 이미지 최적화 없음
- AI 응답 스트리밍 미지원 (면접 질문 생성 시 전체 대기)

## 4. 이번 작업에서 수정한 사항

### 4.1 헤더 메뉴 순서 변경
**변경 전:** 대시보드 - 고객센터 - 구독 - 면접 - 학습 - 도서
**변경 후:** 대시보드 - 면접 - 학습 - 도서 - 고객센터 (모바일 메뉴도 동기화)

### 4.2 구독 메뉴 위치 이동
헤더 상단 메뉴에서 제거 → 프로필 드롭다운 메뉴의 로그아웃 바로 위로 이동

### 4.3 주문 취소/환불 프로세스 개선
**변경 전:** 취소/환불 버튼 → 즉시 CANCELLED/REFUND_REQUESTED
**변경 후:**
1. 취소/환불 버튼 → 사유 선택 모달 (5가지: 단순변심, 주문실수, 결제실수, 상품추가, 기타)
2. 사유 선택 → CANCEL_REQUESTED/REFUND_REQUESTED 상태로 전환
3. 관리자 페이지에서 확인 → 취소 승인/환불 승인 버튼으로 최종 처리
4. 백엔드에 CANCEL_REQUESTED 상태 추가, AdminService에 상태 전이 규칙 추가

### 4.4 카카오 로그인 UI 추가
- 로그인 페이지에 카카오 로그인 버튼 추가
- 카카오 OAuth 콜백 페이지 생성 (KakaoCallbackPage.jsx)
- 라우터에 /auth/kakao/callback 경로 등록
- **주의:** 백엔드 `/api/auth/kakao` 엔드포인트는 아직 미구현 (별도 작업 필요)

### 4.5 프론트엔드 .env 수정
- 중복/오타 VITE_API_BASE_URL 정리

### 4.6 기타
- 로딩 스피너 색상 indigo → mentor 토큰으로 통일
- ORDER_STATUS_LABELS에 CANCEL_REQUESTED 추가
- AdminPage의 배송관리에 CANCEL_REQUESTED/REFUND_REQUESTED 상태 전이 추가

## 7. 결론

이 프로젝트는 **아이디어와 아키텍처 설계는 훌륭하지만, 상용 서비스에는 아직 멀었습니다.**

**강점:**
- 도메인 기반 패키지 구조가 깔끔하고 확장에 유리
- mentor-* 디자인 시스템이 전체 UI에 일관되게 적용
- AI 서버 분리 구조(FastAPI + Spring Boot 프록시)가 현실적
- 면접/학습 핵심 루프가 실제 동작
- 코드 품질과 한글 Javadoc이 매우 좋음

**치명적 약점:**
- 결제가 완전히 Mock (클라이언트가 결제 성공을 직접 선언)
- 소셜 로그인 백엔드 미구현
- 구독이 기능 제한과 연결되지 않음
- 벡터 DB 없이 전체 문서를 프롬프트에 넣는 구조는 비용/품질 모두 비효율적

**42점의 의미:** 학교 프로젝트/포트폴리오로는 80점급이지만,
실제 사용자에게 돈을 받는 상용 서비스 기준으로는 결제/보안/안정성이
모두 미달이라 42점입니다.
